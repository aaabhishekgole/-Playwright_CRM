# Gadget Seva Hub Test Scenarios

## Scope

This document covers the live admin portal, public runner portal, browser runner inbox, and hybrid mobile runner flow.

When external SMS or WhatsApp providers are not configured, validate notification creation through the database, API response, and rider inbox behavior.

## 1. Authentication And Session Handling

### Scenario 1.1 Admin login

Steps

1. Open `/login`
2. Sign in with a valid admin account
3. Verify redirect to dashboard

Expected result

- JWT token is stored
- dashboard loads without `401` or `403`
- success popup/toast appears

### Scenario 1.2 Stale session handling

Steps

1. Manually expire or invalidate the stored token
2. Open a protected route such as `/workspace/pickup-management/assign-pickup`

Expected result

- API rejection is handled cleanly
- user is redirected back to login
- error popup or clear feedback is shown instead of a silent empty module

## 2. Claim Registration

### Scenario 2.1 Register a new claim

Steps

1. Open `Service Requests -> Create Request`
2. Enter customer, device, issue, loan, COI, branch, and employee details
3. Submit

Expected result

- request is created in `service_requests`
- initial status is `REQUEST_CREATED`
- `status_history` contains a creation entry
- success popup appears

### Scenario 2.2 INR formatting

Expected result

- all monetary values across request, billing, and related modules are displayed in INR with rupee formatting

## 3. Pickup Runner Onboarding

### Scenario 3.1 Onboard a pickup runner

Steps

1. Open `Pickup Management -> Runner Onboarding`
2. Enter full name and mandatory mobile number
3. Optionally enter a WhatsApp number
4. Submit

Expected result

- runner user record is created in `users`
- role is `PICKUP_AGENT`
- runner appears in `GET /api/users?role=PICKUP_AGENT`
- runner appears in the `Assign Pickup` dropdown
- success popup appears

### Scenario 3.2 Validation for missing mobile number

Expected result

- submission is blocked
- validation message or error popup clearly explains that mobile number is required

## 4. Pickup Assignment Flow

### Scenario 4.1 Assign pickup from `REQUEST_CREATED`

Steps

1. Open `Pickup Management -> Assign Pickup`
2. Select a request in `REQUEST_CREATED`
3. Choose a pickup runner
4. Enter schedule and notes
5. Submit

Expected result

- request moves to `PICKUP_ASSIGNED`
- `pickups` row is created or updated
- runner token and runner link are generated
- success popup appears

### Scenario 4.2 Rider-only notification targeting

Expected result

- SMS is logged for the assigned rider mobile number only
- WhatsApp is logged for the assigned rider WhatsApp number only
- app notification is created only for the assigned rider account
- a different rider should not see that assignment in `/api/mobile/runner/notifications`

## 5. Runner Smart Link And Browser Flow

### Scenario 5.1 Smart link entry

Steps

1. Open the runner assignment link generated by pickup scheduling

Expected result

- `/runner-access/:token` loads
- on desktop browser it offers browser flow
- on mobile it tries app deep link first and falls back to browser if app is unavailable

### Scenario 5.2 Runner browser inbox

Steps

1. Open `/runner-app`
2. Sign in as the assigned pickup runner

Expected result

- only notifications for that rider are shown
- rider can open the same pickup portal from the inbox
- sign-in and refresh actions show success/error popup feedback

## 6. Runner Pickup Execution

### Scenario 6.1 Accept pickup

Steps

1. Open `/runner-portal/:token`
2. Accept the pickup

Expected result

- pickup acceptance is stored in `pickups.accepted_at`
- request moves into pickup execution flow
- customer and admin notifications are logged
- success popup appears

### Scenario 6.2 Mandatory 10 pickup photos

Steps

1. Upload fewer than 10 required pickup photos
2. Try to complete the pickup

Expected result

- completion is blocked
- clear validation feedback is shown

### Scenario 6.3 Runner customer doorstep updates

Steps

1. Open `/runner-portal/:token`
2. Submit one of these statuses from the shared runner flow:
3. `CUSTOMER_NOT_AVAILABLE`
4. `CUSTOMER_RESCHEDULED`
5. `CUSTOMER_NOT_CONTACTABLE`

Expected result

- request status updates immediately to the selected pickup customer-update status
- pickup acceptance timestamp is stored if it was not already captured
- customer and admin notification records are logged
- case appears in `Pickup Failed Cases`
- case can be reassigned later from `Assign Pickup`

### Scenario 6.4 Optional extra photos

Steps

1. Upload the 10 required photos
2. Upload one or more extra photos
3. Complete pickup

Expected result

- required evidence and optional extras are saved in `attachments`
- pickup completes successfully
- request reaches `PICKUP_COMPLETED`
- customer and admin completion notifications are logged
- success popup appears

- success popup appears

### Scenario 9.2 Approve estimate

- success popup appears

## 10. Repair And Quality Check

### Scenario 10.1 Repair flow

Expected result

- request moves to `REPAIR_IN_PROGRESS`
- then to `REPAIR_COMPLETED`
- repair queues update correctly

### Scenario 10.2 QC flow

Expected result

- repaired device appears in pending QC
- QC pass updates the request to `READY_FOR_DISPATCH`
- QC fail moves the request back to `REPAIR_IN_PROGRESS`
- QC-failed and rework boards reflect the same request history correctly

## 11. Delivery Flow

### Scenario 11.1 Assign delivery

Steps

1. Open `Delivery -> Assign Delivery`
2. assign final-mile delivery

Expected result

- request moves to `DELIVERY_ASSIGNED`
- delivery schedule is stored
- success popup appears

### Scenario 11.2 Delivery completion

Expected result

- request can move to `OUT_FOR_DELIVERY`
- then `DELIVERED`

### Scenario 11.3 Delivery failure and reassignment

## 13. Notifications

### Scenario 13.1 Pickup assignment notifications

Expected result

### Scenario 13.2 Customer and admin notifications

Expected result

## 14. Popups, Toasts, And Alerts

Verify that submit-driven actions show immediate and context-appropriate feedback across modules:

- login
- claim registration
- pickup runner onboarding
- assign pickup
- estimate approval
- cashless approval
- delivery tracking
- payment reconciliation
- attachment upload and delete
- runner portal accept and complete
- runner browser inbox sign-in, refresh, and logout
- hybrid runner app sign-in, refresh, and logout

Expected result

- success actions show clear confirmation
- failures show actionable error wording
- the user is never left guessing whether a submit action worked

## 15. Data Integrity Checks

After a full end-to-end run, verify:

- `service_requests` has the final correct status
- `status_history` reflects every major transition
- `pickups` has schedule, token, acceptance, and completion timestamps
- `attachments` contains required pickup evidence
- `notifications` contains rider, customer, and admin communication records
- `invoices` and `payments` reflect INR billing state
- request detail API returns the same data shown in the portal
